# 설치가이드 — gesture_kiosk 추론 (윈도우, 헤드트래커)

다른 윈도우 PC에 그대로 옮겨 실행하기 위한 가이드입니다.
명령은 전부 **이 폴더(gesture_kiosk) 안에서** 실행합니다.

| 항목 | 기준 |
|---|---|
| OS | Windows 10/11 64비트 |
| GPU | 불필요 (2026-07-18 확정 — 아래 참고) |
| Python | **3.11.5** (설치 시 "Add python.exe to PATH" 체크) |
| 카메라 | USB 웹캠 1대 (오토포커스 권장) |
| 마이크 | 불필요 (입력은 헤드트래커뿐) |

> ⚠ 2026-07-18 전면 개편: 판정 엔진을 팔 쓸기(RTMPose 포즈)에서 **헤드트래커**
> (MediaPipe FaceLandmarker — 코끝 포인터·입 벌리기/응시 클릭·입 오므리기 뒤로가기)로
> 교체했습니다. FaceLandmarker는 CPU만으로 23~27 FPS가 나와(실측) GPU·CUDA
> 의존을 전부 제거했습니다 — 설치가 이전보다 훨씬 단순합니다.

---

## A. 인터넷이 되는 PC — 3단계

```bat
:: 1) 이 폴더를 통째로 복사해 온 뒤, 폴더 안에서:
install.bat

:: 2) 실행 (브라우저가 http://localhost:5000 을 열면 데모 화면)
run.bat
```

`install.bat`이 하는 일: Python 3.11 확인 → 가상환경(venv_win) 생성 →
고정 버전 패키지 설치 → 얼굴 랜드마크 모델 다운로드(face_landmarker.task,
약 4MB) → 설치 검증(smoke_test).

> 📄 **라이선스**: 판정 엔진은 MediaPipe(Apache-2.0) 단일 모델입니다 —
> 저작자 표시 등 고지 의무가 없습니다. 구 HaGRID(YOLOv10, AGPL 리스크)·
> RTMPose 엔진은 2026-07-18부로 완전히 제거됐습니다.

## B. 내부망(인터넷 차단) PC — 번들 반입

정부 민원발급기처럼 인터넷이 없는 PC는, **인터넷 되는 윈도우 PC에서 번들을 만들어**
통째로 반입합니다.

```bat
:: [인터넷 되는 윈도우 PC — 같은 Python 3.11]  이 폴더 안에서:
make_offline_bundle.bat
::  → wheelhouse\ (파이썬 패키지 휠 전부) + models\weights\face_landmarker.task 생성

:: 프로젝트 폴더 전체를 zip → USB/반입절차로 대상 PC에 복사

:: [대상 내부망 PC]  폴더 안에서 (wheelhouse가 있으면 자동으로 오프라인 설치):
install.bat
run.bat
```

> 번들 제작 PC와 대상 PC는 **같은 Python 3.11 / 64비트 윈도우**여야 합니다.
> (pip이 제작 PC 기준으로 휠을 받기 때문). 얼굴 랜드마크 모델(.task)은 고정
> 파일이라 rtmlib 캐시처럼 특수 처리가 필요 없이 `models/weights/`를 그대로
> zip에 포함하면 됩니다.

## D. 회사 키오스크 프로그램(UI) 연동 계약

엔진과 UI는 아래 3개 통로로만 만납니다. 자세한 예시는 `demo_ui/index.html` 참고.

| 방향 | 통로 | 내용 |
|---|---|---|
| 엔진→UI(이산 이벤트) | `event_output` (udp: JSON) 또는 `/data` 폴링 `events` | `select`(입 벌리기/응시) · `go_back`(양 눈 감기 — 뒤로·처음으로 겸용) · `recenter`(코 찡그리기 — 커서 재정렬) |
| 엔진→UI(연속 커서) | `/data` 폴링 `status.cursor_x_ratio`/`cursor_y_ratio` | 데모 UI 전용 — 0~1 화면 비율, 추적 끊기면 `null`. **UDP 계약에는 실리지 않음** (연속 좌표를 이벤트 로그에 매 프레임 넣지 않기 위함) |
| UI→엔진 | `POST /announce {"text": "..."}` | 커서가 올라간 항목 등 화면 안내를 음성으로 읽어달라는 요청 |

- 이동은 좌/우/상/하 이산 이벤트가 아니라 **연속 커서**입니다 — 어떤 항목 위에 있는지는
  UI가 화면 좌표와 커서 좌표를 대조해 판단합니다(호버). `demo_ui/index.html`의
  `updateHover()` 참고.
- 회사 UI가 실시간 커서(호버 인터랙션)를 쓸지, 다른 상호작용 모델을 쓸지는 미정입니다 —
  **UDP로 연속 커서를 전달할지는 별도 협의 필요**(TODO 참고). 그때까지는 discrete
  `select`/`go_back`만 UDP 계약에 포함됩니다.
- 새 수신 규격(소켓/시리얼 등)이 정해지면 `src/pipeline/event_sender.py`에
  Sender 1개를 추가하고 config `event_output.mode`로 선택합니다 — 파이프라인 수정 불필요.

## E. 동작 기준 (2026-07-21 최신 — 커서 재정렬 제스처 추가, 판정은 평상시 기준선 상대값)

| 동작 | 판정 | 이벤트 |
|---|---|---|
| 포인터 이동 | 코끝 위치 — 잠금 직후 캘리브레이션한 중심 기준, 안구간거리로 정규화 | (이벤트 없음 — `/data`·비디오 오버레이의 연속 상태) |
| 선택 (빠름) | 입 벌리기 (jawOpen이 "평상시 기준선 + 여유값"을 넘으면) | `select` |
| 선택 (대체) | 커서가 반경 안에 **1.5초** 머무름(응시) — 마스크 등으로 입 인식이 어려운 사용자용, 항상 함께 활성 | `select` |
| 뒤로(단계별) · 처음으로 | **양 눈 감고 0.6초 버티기** (eyeBlinkLeft/Right 둘 다 "평상시 기준선 + 여유값" 이상 유지) | `go_back` |
| 커서 재정렬 | **코 찡그리기** (noseSneerLeft/Right 중 큰 쪽이 "평상시 기준선 + 여유값"을 넘으면) — 커서 중심을 지금 고개 위치로 다시 잡는다 | `recenter` |
| 사용자 고정 | 초점 맞은(선명·큰) 얼굴 기준 잠금 — 다른 사람 무시 | (이벤트 없음 — 상태) |

> ⚠ 커서 재정렬(2026-07-21 신설): 자리 이동(잠금 해제→재잠금) 없이 커서 중심을 다시
> 잡을 방법이 없던 게 접근성 문제였다(이동이 어려운 사용자에게 특히). 코 찡그리기는
> 입 벌리기·눈 감기와 다른 근육이라 두 동작과 겹치지 않는다. 발화 시 `select`/`go_back`
> 판정 기준선은 그대로 두고(재캘리브레이션 불필요한 부분까지 건드리지 않음) 커서
> 중심만 다시 캘리브레이션한다 — `src/postprocess/head_tracker.py` 참고.
>
> ⚠ 제스처 변천(2026-07-21 2차): 원안 **양 볼 부풀리기**(cheekPuff)는 실기에서
> 인식이 거의 안 됐다(사용자 실측) — MediaPipe 블렌드셰이프 모델이 이 신호를 약하게
> 낸다고 추정. **코 찡그리기**(noseSneerLeft/Right)로 교체하면서, go_back(양쪽 다
> 필요·min)과 반대로 **좌우 중 큰 쪽만**(max) 넘으면 인정하도록 해 카메라 각도·조명
> 편차로 한쪽 신호가 약해도 인식되게 했다 — 오발화보다 미인식이 더 큰 문제인
> 제스처라 방향을 다르게 잡았다.

> ⚠ 뒤로가기 제스처 변천: 매뉴얼 원안 **눈썹 올리기** → 앞머리(bangs)가 눈썹을 가리는
> 사용자는 인식 불가(사용자 지적) → **입 오므리기**로 교체 → select(입 벌리기)와 같은
> 입 근육이라 동작 전환 중 신호가 겹쳐 오작동(사용자 지적) → 입과 완전히 다른 부위인
> **양 눈 감기**로 정착(2026-07-18).

- 입 벌리기·응시는 동시에 조건을 만족해도 `select`가 **1개만** 확정됩니다
  (`head_tracker.click.min_interval_sec` 공용 쿨다운).
- **2026-07-20 정확도 개선**: 입벌림·눈감김 판정을 고정 임계값에서 **"잠금 직후
  캡처한 평상시 표정 기준선 + 여유값(margin)"**으로 바꿨습니다. 실기 관찰 결과
  사람마다 평상시 eyeBlink 값이 0.1~0.6까지 편차가 커서(얼굴 형태·조명·카메라 각도
  영향) 고정 임계 하나로는 어떤 사용자는 오탐, 어떤 사용자는 미탐이 났습니다.
  이제는 코 위치 캘리브레이션과 정확히 같은 시점(`head_tracker.calibration_window_sec`)에
  입/눈 평상시 값도 함께 잡아 기준으로 삼습니다 — 코드는
  `src/postprocess/head_tracker.py`의 `_MedianCalibrator` 참고.
  튜닝은 `head_tracker.mouth_click.open_margin`/`close_margin`,
  `head_tracker.eye_close_cancel.close_margin`에서 — 계기판(화면 하단·비디오 오버레이)에
  현재값과 "기준"값이 함께 표시되니 그 차이를 보고 margin을 조정하면 됩니다.
- 포인터 민감도(`head_tracker.pointer.sensitivity_x`/`sensitivity_y`)도 실기에서 3.0이
  미세한 움직임에도 화면 끝까지 튀어 0.8/0.5로 낮췄습니다(상하는 고개 회전 범위가
  좌우보다 좁아 더 낮게) — 카메라 거리·설치 위치가 달라지면 재조정하세요.
- 캘리브레이션(`head_tracker.calibration_window_sec`)은 잠금 직후 코 위치·입벌림·
  눈감김의 **중앙값**(평균 아님)으로 기준을 정합니다 — 과도기 프레임 1~2장이 섞여도
  기준점이 흔들리지 않게 하기 위함입니다(2026-07-18 실기: 평균일 때 커서가 상단에
  눌러붙는 현상이 있었습니다).
- **뒤로가기는 단계별**입니다(2026-07-20) — 증명서 선택→번호 입력→장수 선택→발급
  확인 흐름에서, 뒤로가기를 하면 바로 전 단계로만 돌아가고 입력값(번호·장수)은
  그대로 남아 있습니다. 발급 완료 화면에서만 예외로 처음(증명서 선택)으로 돌아갑니다
  (더 되돌아갈 입력 단계가 없으므로).
- 스크롤(응시 1초로 화면 아래로) 매뉴얼에 언급됐으나 현재 데모 UI가 고정 그리드라
  v1 범위 밖입니다(config `head_tracker.scroll.enabled: false`).

## E-1. 카메라 설치 기준 (헤드트래커 — 2026-07-18)

| 항목 | 기준 | 이유 |
|---|---|---|
| 사용자 거리 | 0.5 ~ 0.9 m (매뉴얼 권장) | 너무 멀면 안구간거리(정규화 자)가 작아져 커서가 둔감해짐 |
| 프레임 구도 | 얼굴 전체 포함 | 코·눈·입 랜드마크가 모두 보여야 함 |
| 카메라 높이 | 사용자 눈높이 근처 | 위/아래 각이 크면 코끝 캘리브레이션 중심이 치우침 |
| 지향 | 사용자 정면 | 측면 각이 크면 안구간거리(정규화 자)가 좁게 잡혀 커서가 과민해짐 |
| 조명 | 밝게, 역광 회피 | 역광이면 얼굴 랜드마크·블렌드셰이프 신뢰도가 떨어짐 |

설치 후 확인: 데모 화면에서 사용자가 표준 위치에 섰을 때 USER LOCK 박스와
코끝 커서(초록 원)가 안정적으로 표시되고, 고개를 자연스럽게 돌렸을 때 커서가
화면 끝까지 부드럽게 따라가는지(즉시 튀지 않는지) 확인하세요.

## F. 개인정보 처리

2026-07-16 주민등록증 OCR 기능이 제거되어 **엔진은 개인정보를 다루지 않습니다** —
입력은 얼굴 랜드마크(좌표 숫자)·블렌드셰이프(표정 점수)뿐이고, 카메라 프레임은
디스크에 저장하지 않으며, 이벤트에는 동작 이름(select/go_back/recenter)만 실립니다.
본인확인 절차는 회사 프로그램 담당.

> 📄 **2026-07-18 신설**: `demo_ui/index.html`에 예시로 주민등록번호 입력 키패드를
> 추가했다(발급 전 본인확인 흐름 시연용). 입력값은 **브라우저 안에만** 머물고
> 백엔드(엔진)로 전송되는 API 호출이 없다 — 엔진은 여전히 select/go_back/recenter
> 이벤트만 받고 입력값의 존재 자체를 모른다. 화면에도 뒷자리는 즉시 마스킹(●)
> 표시된다. 실제 본인확인 절차·수집 근거는 여전히 회사 프로그램 담당(위 원칙 유지).

## G. 문제 해결

| 증상 | 조치 |
|---|---|
| `install.bat`에서 Python을 못 찾음 | Python 3.11.5 설치 시 PATH 체크 누락 — 재설치 또는 `py -3.11` 확인 |
| 카메라가 안 열림/수십 초 걸림 | config `camera.device_id` 0→1 변경, `camera.windows_backend`를 msmf↔dshow로 바꿔 시도 |
| 캡처 FPS가 ~5에 묶임 | `windows_backend: msmf` 확인 (dshow는 카메라에 따라 무압축 YUY2로만 협상돼 USB 한계에 걸림) |
| 한글 음성이 안 나옴 | 윈도우 설정 → 시간 및 언어 → 음성 → 한국어 음성(Heami 등) 설치 |
| 커서가 상단/한쪽에 눌러붙어 잘 안 움직임 | config `head_tracker.pointer.sensitivity_x`/`sensitivity_y`를 낮춘다 (기본 0.8/0.5). 캘리브레이션이 이상 프레임에 걸렸을 수도 있으니 재잠금(자리 이탈 후 복귀)도 시도 |
| 커서가 미세한 움직임에도 화면 끝까지 튐 | config `head_tracker.pointer.sensitivity_x`/`sensitivity_y`를 낮춘다 |
| 눈을 안 감았는데 자꾸 뒤로 가짐 | config `head_tracker.eye_close_cancel.close_margin`을 올린다 (기본 0.3) — 계기판의 "기준"값과 현재값 차이를 보고 조정 |
| 눈을 감아도 뒤로가기가 안 됨 | config `head_tracker.eye_close_cancel.close_margin`을 낮춘다 |
| 입을 안 벌렸는데 자꾸 선택됨 | config `head_tracker.mouth_click.open_margin`을 올린다 (기본 0.35) |
| 응시(1.5초)해도 선택 안 됨 | config `head_tracker.dwell_click.radius_ratio`를 넉넉히 올린다 (기본 0.04) |
| 다른 사람 얼굴에 반응 | `person_lock.sharpness_weight`를 올려 초점 선명도 비중을 높인다 |
| 추론 FPS가 30에 못 미침 | 시연 화면(브라우저 창)을 한 개만 연다 — 창마다 영상 인코딩이 따로 돌아 부하가 배가된다(2026-07-20 실측). 그래도 부족하면 `model.infer_scale_ratio` 0.75→0.5 |
| 커서가 화면 항목과 어긋남 | `camera.proc_width_px/proc_height_px`가 16:9인지 확인 — 데모 UI 스테이지와 종횡비가 다르면 가장자리에서 어긋난다 |
